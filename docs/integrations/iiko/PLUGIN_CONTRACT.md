# Revisitr × iiko: HTTP-контракт плагина

> Версия: 0.2 — MVP
> Статус: спецификация для реализации (бэкенд строится по ней)
> Дата: июнь 2026

Контракт между плагином iikoFront (`.dll`) и бэкендом Revisitr. Описывает ровно
тот дизайн, под который реализуется бэкенд. Парный документ — `PLUGIN_DESIGN.md`
(устройство плагина). Бизнес-сценарии — `revisitr_usecase_iiko.md`.

---

## 1. Общее

- Транспорт: **JSON over HTTPS**. Только HTTPS, без plaintext-фолбэка.
- Базовый путь: **`/api/v1/pos-plugin`**.
- Тело запроса и ответа — `application/json; charset=utf-8`.
- Все суммы — в минимальной валютной единице программы (по умолчанию рубли,
  целые числа). Поле `currency` возвращается в `/identify` и `/config`.
- Время на стороне бэкенда — UTC; плагин временем не оперирует.

---

## 2. Аутентификация

Каждый запрос плагина несёт заголовок с **ключом заведения**:

```
X-API-Key: rvk_live_8f3c9a21b7e44d0fa1c6e2d5b9874f30
```

- Ключ выдаётся в веб-кабинете Revisitr при подключении заведения и вставляется
  в настройки плагина один раз (см. `PLUGIN_DESIGN.md`, раздел «Конфигурация»).
- Бэкенд **не хранит сам ключ** — только его `SHA-256`-хеш в таблице
  `pos_plugin_keys`. При запросе бэкенд хеширует пришедший ключ и ищет
  совпадение по хешу.
- Найденная запись отображает ключ на **`org_id` + `integration_id`** — это
  определяет заведение, программу лояльности и контекст идемпотентности
  (все транзакции скоупятся по `integration_id`).
- Отсутствующий или неизвестный ключ → `401 Unauthorized`,
  `{"error": "invalid api key"}`.
- Ротация ключа — выпуск нового хеша в кабинете; старый деактивируется.
  Ключ нигде не логируется (см. `PLUGIN_DESIGN.md`, «Безопасность»).

### Rate-limiting

`POST /identify` ограничен по частоте на уровне ключа заведения (защита от
перебора кодов: код — короткое русское слово, теоретически подбираем).
Превышение лимита → `429 Too Many Requests`, `{"error": "rate limited"}`.
Плагин в ответ на `429` не зацикливается на ретраях, а показывает кассиру
ошибку и даёт повторить вручную. `redeem`/`accrue` лимитируются мягче —
они идемпотентны и привязаны к закрытию заказа.

## 3. Идентификация гостя

Гость предъявляет один из двух идентификаторов; оба попадают в одно поле `code`
запроса `/identify`:

| Способ | Что предъявляет гость | Как попадает в плагин |
|--------|----------------------|----------------------|
| **QR/штрихкод** (основной) | QR из Telegram-бота | стационарный сканер → событие `OrderEditBarcodeScanned` |
| **Короткий код** (ручной фолбэк) | одно русское слово ≥ 5 букв, рядом с QR в боте | кассир вводит вручную в поле диалога |

### Короткий код: правила

- Одно русское слово **длиной ≥ 5 букв** (выдаётся ботом, человекочитаемо и
  произносимо вслух).
- **Одноразовый**: инвалидируется сразу после успешного `/identify`.
- **TTL ~5 минут**: по истечении бот выдаёт новый.
- В один момент у гостя активен **только один** код (выпуск нового гасит старый
  и удаляет предыдущее сообщение с кодом).
- Нормализация перед сравнением на бэкенде:
  - регистронезависимо (`КОФЕ` == `кофе`);
  - `ё` → `е`;
  - `й` → `и`.
  Кассиру не нужно угадывать точное написание — `подъезд`/`подьезд`,
  `ёлка`/`елка` сматчатся.
- QR несёт непрозрачный токен и сравнивается точно (без нормализации).

Бэкенд по `code` находит ровно одного гостя в рамках `org_id` ключа. Гость не
найден / код истёк / уже использован — см. таблицу ошибок (раздел 7).

## 4. Эндпоинты

Все примеры приведены с заголовком `X-API-Key` (опущен в JSON для краткости).

### 4.1 `POST /identify`

Потребляет код гостя, открывает короткоживущую сессию для последующих
`redeem`/`accrue`. Возвращает профиль гостя и потолок списания для этого заказа.

**Запрос**

```json
{
  "code": "кофе",
  "order_total": 1200
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `code` | string | QR-токен или короткий код (нормализуется для короткого кода) |
| `order_total` | int | Текущая сумма заказа, нужна для расчёта `available_to_redeem` |

**Ответ `200 OK`**

```json
{
  "session": "ps_3b9d2f17c8a14e6b",
  "client": {
    "id": "cl_91a0",
    "name": "Иван П.",
    "balance": 320,
    "currency": "RUB",
    "available_to_redeem": 160
  }
}
```

| Поле | Описание |
|------|----------|
| `session` | токен сессии, **TTL ~15 минут**, используется в `redeem`/`accrue` |
| `client.balance` | текущий баланс бонусов |
| `client.available_to_redeem` | `min(balance, order_total * max_redeem_percent / 100)` — потолок списания для этого заказа |

- `available_to_redeem` рассчитывается бэкендом по правилам программы
  (`max_redeem_percent` из `/config`); плагин его не пересчитывает, а показывает
  кассиру как «Доступно к списанию».
- Успешный `/identify` **потребляет код** (одноразовость) и **создаёт сессию**.
- Скоуп идемпотентности `redeem`/`accrue` — пара `(integration_id, order_id)`,
  где `integration_id` берётся из ключа, поэтому он в теле не передаётся.

**Ошибки**: `401`, `404` (гость/код не найден), `409` (код истёк/использован),
`429` (rate limit), `503` (бэкенд недоступен). См. раздел 7.

---

### 4.2 `POST /redeem`

Списывает бонусы с баланса гостя. **Идемпотентен** по
`(integration_id, order_id, "redeem")`.

**Запрос**

```json
{
  "session": "ps_3b9d2f17c8a14e6b",
  "order_id": "iiko-order-7f2a9c01",
  "amount": 160
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `session` | string | токен из `/identify` |
| `order_id` | string | id заказа iiko — ключ идемпотентности |
| `amount` | int | сумма списания, `0 < amount ≤ available_to_redeem` |

**Ответ `200 OK`**

```json
{
  "ok": true,
  "balance_after": 160
}
```

- Частичное списание допустимо (`amount` < `balance`).
- `amount` > `available_to_redeem` → `400`,
  `{"error": "amount exceeds available"}`.
- Недостаточно бонусов → `400`, `{"error": "insufficient balance"}`.
- Повторный вызов с тем же `order_id` возвращает **исходный** результат
  (`balance_after` тот же), баланс не списывается дважды (см. раздел 6).

**Ошибки**: `400`, `401`, `404` (сессия не найдена/истекла), `503`.

---

### 4.3 `POST /accrue`

Начисляет бонусы за визит. **Идемпотентен** по
`(integration_id, order_id, "accrue")`.

**Запрос**

```json
{
  "session": "ps_3b9d2f17c8a14e6b",
  "order_id": "iiko-order-7f2a9c01",
  "amount": 1040
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `session` | string | токен из `/identify` |
| `order_id` | string | id заказа iiko — ключ идемпотентности |
| `amount` | int | база начисления (обычно сумма, не покрытая бонусами) |

**Ответ `200 OK`**

```json
{
  "ok": true,
  "accrued": 104,
  "balance_after": 264
}
```

- `accrued` рассчитывается бэкендом: `amount * accrual_percent / 100`
  (правила — на стороне Revisitr).
- Повторный вызов с тем же `order_id` возвращает исходный результат, повторного
  начисления нет.
- После `accrue` Telegram-бот шлёт гостю уведомление об изменении баланса
  (на стороне Revisitr, плагина не касается).

**Ошибки**: `400`, `401`, `404`, `503`.

---

### 4.4 `GET /config`

Возвращает правила программы лояльности заведения. Плагин дёргает его при
старте, чтобы изменения правил в кабинете применялись **без переустановки**
`.dll`.

**Ответ `200 OK`**

```json
{
  "program_type": "bonus",
  "currency": "RUB",
  "max_redeem_percent": 50,
  "accrual_percent": 10,
  "sum_with_iiko_discounts": true
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `program_type` | string | `bonus` \| `discount` |
| `currency` | string | код валюты программы |
| `max_redeem_percent` | int | потолок списания, % от суммы чека |
| `accrual_percent` | int | % начисления от базы |
| `sum_with_iiko_discounts` | bool | суммировать ли бонусы со скидками iiko (happy hour и т.п.) |

**Ошибки**: `401`, `503`.

---

### 4.5 `POST /order`

Передаёт каждый чек после закрытия в iikoFront. Вызов не влияет на оплату и
никогда не блокирует кассира: при ошибке плагин только логирует её.

**Запрос**

```json
{
  "order_id": "iiko-order-7f2a9c01",
  "source": "hall",
  "ordered_at": "2026-07-20T12:30:00Z",
  "total": 1250,
  "table_num": "12",
  "waiter_name": "Иван",
  "items": [
    {
      "name": "Капучино",
      "quantity": 2,
      "price": 350
    }
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `order_id` | string | ID закрытого заказа iiko; ключ идемпотентности |
| `source` | `hall` \| `delivery` | Тип чека в iikoFront |
| `ordered_at` | RFC3339 datetime | Время закрытия чека в UTC |
| `total` | number | Итоговая сумма чека |
| `table_num` | string | Стол, если есть |
| `waiter_name` | string | Официант, если есть |
| `items` | array | Блюда с названием, количеством и ценой за единицу |

**Ответ `204 No Content`**

- Бэкенд сохраняет источник `hall` или `delivery`.
- Повторная отправка того же `order_id` идемпотентна.
- Если этот же чек позже пришёл как iikoCloud delivery, бэкенд объединяет его
  в одну запись по `(integration_id, external_id)`, сохраняя данные кассы.

**Ошибки**: `400`, `401`, `500`. Плагин не показывает их кассиру и не отменяет
закрытие чека.

---

## 5. Формат ошибок

Все ошибки — единое тело:

```json
{ "error": "human-readable reason" }
```

HTTP-статус несёт семантику (раздел 7); строка `error` — для лога/диагностики,
кассиру плагин показывает свой локализованный текст.

---

## 6. Идемпотентность

Деньги. Повторная доставка запроса (ретрай по таймауту, дубль события закрытия
заказа в iikoFront, переотправка после обрыва сети) **не должна** ни списать,
ни начислить бонусы дважды.

- **Ключ идемпотентности** — `order_id` (id заказа iiko) в скоупе
  `(integration_id, operation)`, где `operation` ∈ {`redeem`, `accrue`}.
  `integration_id` берётся из ключа заведения.
- Бэкенд хранит результат первой успешной операции по этому ключу. Повтор с тем
  же `order_id` возвращает **тот же** ответ (`balance_after`, `accrued`) и тот
  же HTTP-статус — без побочного эффекта.
- `redeem` и `accrue` — независимые операции с разными `operation`, поэтому по
  одному `order_id` корректно идут оба ровно по одному разу.
- Плагин обязан использовать **один и тот же `order_id`** при ретраях
  (см. `PLUGIN_DESIGN.md`, «Устойчивость»). Это делает любой ретрай безопасным.
- `/identify` идемпотентным **не является** намеренно: код одноразовый, повтор
  по уже потреблённому коду → `409` (раздел 7).

## 7. Ошибки и edge-cases

Маппинг сценариев из `revisitr_usecase_iiko.md` (раздел 8) на HTTP-статусы и
поведение плагина:

| Ситуация | Статус | Тело `error` | Поведение плагина |
|----------|:------:|--------------|-------------------|
| Гость не найден (несуществующий код/QR) | `404` | `guest not found` | «Гость не найден». Кассир закрывает заказ штатно, без бонусов. |
| Код истёк | `409` | `code expired` | «Код недействителен. Попросите гостя обновить код в боте». |
| Код уже использован (повтор) | `409` | `code already used` | То же: код одноразовый, нужен новый. |
| Недостаточно бонусов | `400` | `insufficient balance` | Показать доступный остаток; возможно частичное списание. |
| Сумма списания > доступного | `400` | `amount exceeds available` | Показать `available_to_redeem`, дать ввести меньшую сумму. |
| Сессия истекла / не найдена | `404` | `session not found` | Заново открыть диалог идентификации. |
| Неверный/просроченный ключ | `401` | `invalid api key` | «Плагин не настроен». Закрыть заказ штатно; владельцу — проверить ключ. |
| Rate limit на `/identify` | `429` | `rate limited` | Показать ошибку, дать повторить вручную; не зацикливать ретраи. |
| Revisitr API недоступен | `503` | `service unavailable` | Показать ошибку. Кассир закрывает заказ **без бонусов**. Постфактумного начисления в MVP нет. |

Частичная оплата (бонусы + наличные/карта) — штатная функция iiko: плагин
применяет бонусный тип оплаты на `amount`, остаток гость гасит другим способом.
Отдельного контракта не требует.

**Принцип MVP**: при любой ошибке плагин **никогда не блокирует кассира** —
заказ всегда можно закрыть штатно. Бонусы — best-effort поверх штатного
закрытия. Резервного/постфактумного начисления в MVP нет.
