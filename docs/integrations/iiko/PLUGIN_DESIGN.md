# Revisitr × iiko: дизайн плагина iikoFront

> Версия: 0.1 — MVP
> Статус: спецификация для реализации (плагин строится по ней)
> Дата: июнь 2026

Устройство плагина iikoFront (`.dll`), который строится на C#/.NET. HTTP-контракт
с бэкендом — `PLUGIN_CONTRACT.md`. Бизнес-сценарии и edge-cases —
`revisitr_usecase_iiko.md`. Этот документ — как плагин устроен и как он
реализует флоу из юзкейса.

---

## 1. Платформа и лицензирование

- **iikoFront Front.Net plugin** — расширение кассового приложения iikoFront на
  платформе Front.Net (`.dll`, .NET Framework 4.7.2, C#).
- Деплой: копирование `.dll` в папку `Front.Net\Plugins\Resto.Front.Api.Revisitr\`
  кассового ПК, перезапуск iikoFront.
- Регистрируется как **отдельный тип оплаты** «Revisitr Бонусы» в экране
  закрытия заказа.
- **Лицензия:** `iikoAPIPayment`, ModuleID **`21016318`** (категория: системы
  лояльности + эквайринг). Бесплатна, не требует регистрации через портал.
  Прописывается в коде атрибутом:
  ```csharp
  [PluginLicenseModuleId(21016318)]
  public sealed class RevisitrPlugin : IFrontPlugin { ... }
  ```
- **Интерфейс платёжной системы:** плагин регистрирует реализацию
  `IExternalPaymentProcessor` через
  `PluginContext.Operations.RegisterPaymentSystem(processor)`.
  Ключевые методы процессора:
  - `CollectData` — вызывается до оплаты; здесь открываем диалог, получаем
    код, вызываем `identify`, сохраняем сессию.
  - `Pay` — применяет оплату бонусами; здесь вызываем `redeem`.
    Отказ — `throw new PaymentActionFailedException("сообщение")`.
    Отмена кассиром — `throw new PaymentActionCancelledException()`.
  - `ReturnPayment` — возврат (MVP: логируем, бонусы не восстанавливаем).
  - Все остальные (`PaySilently`, `EmergencyCancelPayment`, …) реализуются
    минимально (NotImplementedException или no-op).

### 1.1 Универсальность: один плагин на много версий iikoFront

Плагин должен работать на кассах разных версий у разных клиентов. Механика iiko
делает это возможным без «плагина под каждую версию»:

- **Контракт-DLL — только на этапе компиляции, НЕ поставляется с плагином.**
  Ссылку на `Resto.Front.Api.Vх` ставим с `Copy Local = False` (для NuGet до
  V7Preview2 — `<PrivateAssets>all</PrivateAssets>`, с V7Preview2+ отключается
  автоматически). В рантайме iikoFront подставляет **свою** реализацию контракта
  из своей папки. Именно это даёт версо-независимость.
  ⚠️ Если по ошибке положить контракт-DLL рядом с плагином — падение
  `TypeLoadException: could not load type 'Resto.Front.Api.Vх.ILog'`. Это грабля №1.

- **Гарантия обратной совместимости (цитата iiko):** «Плагин, разработанный под
  определённую версию API, будет одинаково работать с любыми версиями iikoRMS,
  поддерживающими эту версию API.» → один бинарь покрывает весь диапазон
  iikoFront, поддерживающих выбранную версию API.

- **Стратегия выбора версии:** наша логика примитивна (тип оплаты + диалог +
  событие закрытия) — есть во всех версиях, к новым фичам не привязаны. Значит
  целимся в **самую низкую ещё живую LTS**. Матрица на момент написания:

  | API | Статус | Диапазон iikoFront |
  |-----|--------|--------------------|
  | V6  | снята с 8.8 | до 8.7 |
  | V8  | obsolete, живёт до 10.3 | широкая полоса + будущее |
  | V9  | current | новые |

  **Целевая версия по умолчанию — V8** (V6 уже мёртв на 8.8+). Финальный выбор
  зависит от распределения версий iikoFront у клиентов — собрать до релиза.

- **Build-matrix для хвостов:** исходник один, различается только версия
  референса на контракт. Тот же проект собирается под V6/V8/V9 → 2-3 `.dll`.
  Это конфигурации сборки, не разные кодовые базы.

- **Кабинет отдаёт нужный бинарь** по версии кассы клиента (её спрашиваем при
  онбординге). Плюс runtime-детект `GetApplicationVersion()` /
  `GetSupportedApiVersions()` для подстройки/защиты.

Шаги установки (из `revisitr_usecase_iiko.md`, раздел 10):

1. Владелец скачивает `.dll` из кабинета Revisitr.
2. Копирует в `Front.Net\Plugins` на кассовом ПК.
3. Перезапускает iikoFront.
4. В iikoOffice добавляет тип оплаты «Revisitr» (по инструкции из кабинета).
5. Вводит API-ключ заведения в настройках плагина.

Подробный гайд с иллюстрациями — в личном кабинете Revisitr.

## 2. Архитектура и компоненты

```
RevisitrPlugin (.dll)
 ├─ PaymentTypeHandler          — тип оплаты «Revisitr Бонусы»
 │    · enter amount → применить как оплату к заказу
 │    · on close → вызвать redeem + accrue
 │
 ├─ IdentifyDialog (WPF/WinForms)  — диалог идентификации/баланса
 │    · [Сканировать QR/штрихкод]
 │    · [Ввести код вручную: _____ ]
 │    · показ баланса и «Доступно к списанию»
 │    · [Списать сумму] / [Только начислить]
 │
 ├─ BarcodeScannedHandler       — подписка на OrderEditBarcodeScanned
 │    · скан сканера → тот же путь identify, что и ручной код
 │
 ├─ RevisitrApiClient (HttpClient)  — вызовы PLUGIN_CONTRACT.md
 │    · X-API-Key из настроек
 │    · identify / redeem / accrue / config
 │
 └─ PluginSettings              — локальный конфиг (API base URL, API key)
```

| Компонент | Назначение |
|-----------|-----------|
| **PaymentTypeHandler** | реализует тип оплаты «Revisitr Бонусы»; хранит выбранную сумму списания и сессию; на закрытии заказа триггерит `redeem` + `accrue` |
| **IdentifyDialog** | UI (WPF или WinForms — что доступно в SDK iikoFront): кнопки `[Сканировать QR]` и `[Ввести код]`, показ профиля/баланса, выбор суммы либо «только начислить» |
| **BarcodeScannedHandler** | подписка на `OrderEditBarcodeScanned`; отсканированный код подаётся в **тот же** `identify`, что и ручной ввод |
| **RevisitrApiClient** | `HttpClient`-обёртка над контрактом; подставляет `X-API-Key`, сериализует JSON, маппит статусы на типизированные ошибки |
| **PluginSettings** | API base URL и API key, вводятся один раз при установке |

Один путь идентификации: и сканер (`OrderEditBarcodeScanned`), и ручной ввод
кода кладут значение в `code` и вызывают `POST /identify`. Дублирования логики
нет.

## 3. Runtime-флоу

Сопоставление с флоу из `revisitr_usecase_iiko.md`.

### 3.1 Списание бонусов (юзкейс 3.1)

1. Кассир в экране оплаты нажимает тип оплаты «Revisitr Бонусы».
2. Открывается `IdentifyDialog`: `[Сканировать QR]` / `[Ввести код]`.
3. Скан (через `OrderEditBarcodeScanned`) или ручной код →
   `POST /identify {code, order_total}` (где `order_total` = текущая сумма
   заказа из iikoFront).
4. Диалог показывает: имя, баланс, «Доступно к списанию» (`available_to_redeem`).
5. Кассир уточняет у гостя сумму и подтверждает её (или жмёт «Списать всё
   доступное»).
6. Плагин применяет тип оплаты «Revisitr Бонусы» на выбранную сумму → строка
   скидки/оплаты в чеке iiko.
7. Кассир закрывает заказ штатно.
8. **После закрытия** плагин вызывает `POST /redeem {session, order_id, amount}`,
   затем `POST /accrue {session, order_id, accrue_base}` (база = сумма, не
   покрытая бонусами).
9. Telegram-бот (на стороне Revisitr) уведомляет гостя об изменении баланса.

### 3.2 Только начисление (юзкейс 3.2)

1–4. Идентификация как в 3.1.
5. Кассир жмёт **[Только начислить]** — сумма списания = 0.
6. Заказ закрывается штатно без применения бонусного типа оплаты.
7. После закрытия плагин вызывает только `POST /accrue` на полную сумму заказа
   (`redeem` не вызывается).

### 3.3 Гость не предъявил карту (юзкейс 3.3)

Кассир закрывает заказ штатно, диалог Revisitr не открывает. Плагин не делает
никаких вызовов. Бонусы не начисляются и не списываются. Постфактумного
начисления в MVP нет.

### 3.4 Сплит счёта (юзкейс 4)

- **Один общий счёт**: один гость идентифицируется, его бонусы применяются ко
  всему чеку — обычный флоу 3.1.
- **Раздельные счета**: кассир штатно разбивает заказ iiko на чеки; для каждого
  чека `IdentifyDialog` открывается заново со своим `order_id`. Идемпотентность
  по `order_id` гарантирует независимость чеков.

### Ключевое решение MVP: redeem+accrue после закрытия

Бэкенд в MVP делает `redeem` **одним атомарным вызовом на момент закрытия**, без
отдельного reserve/confirm. Поэтому плагин:

- на шаге выбора суммы **не** резервирует бонусы заранее;
- вызывает `redeem` и `accrue` **только после того, как заказ закрыт** в
  iikoFront;
- при отмене/void заказа до закрытия — **не вызывает** `redeem`/`accrue` вовсе
  (бонусы не тронуты). Отдельного `cancel`-вызова в MVP нет; если в будущем
  появится reserve/confirm — добавится и `cancel`.

## 4. Устойчивость

Плагин **никогда не блокирует кассира**. Бонусы — best-effort поверх штатного
закрытия заказа.

- **Таймауты**: все HTTP-вызовы с коротким таймаутом (ориентир: `identify` ~5 с,
  `redeem`/`accrue` ~5 с). Превышение трактуется как недоступность бэкенда.
- **Ретраи**: `redeem`/`accrue` ретраятся (1–2 раза, backoff) с **тем же
  `order_id`**. Идемпотентность контракта (см. `PLUGIN_CONTRACT.md`, раздел 6)
  делает ретрай безопасным — двойного списания/начисления не будет.
- **Offline / fail closed**: если бэкенд недоступен на этапе `identify` —
  показать ошибку, дать закрыть заказ **без бонусов**. Если недоступен на этапе
  `redeem`/`accrue` после исчерпания ретраев — заказ уже закрыт штатно, бонусы
  не проведены (постфактумного начисления в MVP нет). Кассир не ждёт сеть.
- **Дубли событий закрытия**: повторный триггер закрытия в iikoFront безопасен —
  тот же `order_id` вернёт исходный результат.

## 5. Безопасность

- **API-ключ** хранится в настройках плагина (`PluginSettings`), вводится один
  раз при установке.
- Только **HTTPS** к бэкенду Revisitr; plaintext недопустим.
- Ключ **никогда не логируется** (ни в файлы, ни в консоль iikoFront, ни в
  диагностику). В заголовок `X-API-Key` подставляется только в момент запроса.
- Бэкенд хранит лишь `SHA-256`-хеш ключа — плагин это знать не обязан, но из
  этого следует: ключ нельзя «восстановить» из бэкенда, при потере — выпускать
  новый в кабинете.
- Коды идентификации **одноразовы** и короткоживущи (TTL ~5 мин на стороне
  Revisitr); плагин их не кеширует и не переиспользует.

## 6. Обновление конфигурации

При старте плагин вызывает `GET /config` и применяет правила программы:
`program_type`, `currency`, `max_redeem_percent`, `accrual_percent`,
`sum_with_iiko_discounts`.

- Вся бизнес-логика — на бэкенде Revisitr; плагин лишь отображает правила и
  суммы, рассчитанные бэкендом.
- Изменение правил в веб-кабинете (например, % списания) применяется **без
  переустановки** `.dll` — достаточно перезапуска iikoFront (повторный
  `GET /config`).
- Если `/config` недоступен при старте — плагин может работать на последнем
  известном конфиге либо запросить конфиг лениво перед первым `identify`;
  при полной недоступности — fail closed (раздел 4).

## 7. Дорожная карта

Зеркалит дорожную карту юзкейса (`revisitr_usecase_iiko.md`, раздел 11):

| # | Задача | Приоритет |
|---|--------|-----------|
| 1 | UI «максимальный % списания» — отдельная подсказка/ограничение в диалоге | Высокий |
| 2 | Поддержка **iikoWaiter** — сканирование QR камерой смартфона официанта у стола | Высокий |
| 3 | **Auto-updater** плагина — автоматическая замена `.dll` без участия клиента | Средний |
| 4 | Настройка совместимости со скидками iiko (суммировать / не суммировать) | Средний |
| 5 | **Постфактумное начисление** (гость пробивает чек сам в боте) — потребует серверной поддержки, не только плагина | Низкий |
