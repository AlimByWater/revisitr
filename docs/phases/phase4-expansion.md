# Фаза 4: Расширение (Expansion)

## 1. Обзор и видение

Фаза 4 — набор пост-MVP модулей для масштабирования Revisitr. Если фазы 1-3 строят ядро продукта (лояльность, управление клиентами, аналитика, интеграции), то фаза 4 превращает Revisitr в полноценную платформу: новые каналы взаимодействия, монетизация, расширенная аналитика.

Ключевые направления:

- **Админ-бот** — управление бизнесом из Telegram без веб-панели
- **Wallet** — лояльность без Telegram, через Apple/Google Wallet
- **Маркетплейс** — трата баллов внутри бота, оплата заказов
- **Биллинг** — тарифы и монетизация платформы
- **Расширенные рассылки** — A/B-тесты, мультиканальность
- **Расширенная сегментация** — предиктивные и поведенческие сегменты

Эта фаза — дорожная карта, а не детальная спецификация. Приоритеты и порядок реализации могут меняться в зависимости от обратной связи от пользователей и бизнес-метрик.

## 3. Подфазы

### 4A. Админ-бот в Telegram

**Цель**: Позволить владельцам и менеджерам бизнеса управлять заведением из Telegram — без открытия веб-панели. Telegram удобен для задач, которые сложно делать в вебе: отправка кружков (видео-сообщений), голосовых, стикеров в рассылках.

**Бизнес-ценность**: Снижение порога входа для владельцев. Быстрые действия на ходу — создать промокод, запустить рассылку, посмотреть статистику за день.

#### Функции

- Создание и отправка рассылок с rich media (кружки, голосовые, стикеры)
- Быстрое создание и активация промокодов
- Управление авто-действиями (старт/стоп)
- Уведомления с базовой аналитикой (дневная/недельная сводка)

#### Архитектура

```
cmd/
├── server/       # API-сервер (админка)         — существует
├── bot/          # Клиентский бот (Telegram)     — существует
└── admin-bot/    # Админ-бот (Telegram)          — НОВЫЙ сервис
```

- **Третий сервис** (`cmd/admin-bot`) — отдельный процесс, отдельный Telegram-бот
- **Общие usecase-ы** с `cmd/server` — та же бизнес-логика (campaigns, promo, loyalty)
- **Аутентификация**: одноразовая привязка Telegram-аккаунта к учётной записи пользователя
  - Генерация одноразового кода в веб-панели → ввод в админ-боте
  - После привязки — извлечение `org_id` и `role` из связки
- **Отдельный telego.Bot** instance с собственным токеном
- **Scheduler**: cron-задачи для отправки дневных/недельных сводок

#### Ключевые решения

- Использовать существующие usecase-интерфейсы — не дублировать логику
- Разграничение прав: owner vs manager (не все менеджеры могут запускать рассылки)
- Хранение медиа: Telegram file_id для кружков/голосовых (не загружать на свой сервер)
- Rate limiting на действия через бота

#### Оценка объёма: **L**

#### Приоритет: **Высокий** — уникальная ценность, сложно реализовать в конкурентных продуктах

### 4B. Интеграция с Wallet (Apple Wallet / Google Wallet)

**Цель**: Карта лояльности без необходимости использовать Telegram-бота. Клиент получает карту в Apple/Google Wallet с балансом, уровнем и QR-кодом.

**Бизнес-ценность**: Расширение охвата — не все клиенты готовы взаимодействовать с ботом, но Wallet-карта привычна и ненавязчива. Номер телефона всегда доступен (обязательное поле при регистрации).

#### Функции

- Генерация `.pkpass` (Apple) и Google Wallet pass для каждого клиента
- Отображение на карте: баланс лояльности, текущий уровень, QR-код
- Push-обновления карты при изменении баланса
- Клиент может использовать Wallet БЕЗ активного Telegram-бота
- Интеграция с существующим loyalty engine (фаза 1)

#### Архитектура

```
internal/
├── usecase/
│   └── wallet/          # Генерация и обновление pass-файлов
├── repository/
│   └── postgres/
│       └── wallet.go    # wallet_passes таблица (client_id, serial, push_token)
└── controller/
    └── http/group/
        └── wallet/      # GET /api/v1/wallet/:serial — скачивание pass
                         # POST /api/v1/wallet/register — push registration
```

- **Apple Wallet**: библиотека для генерации `.pkpass` (подпись сертификатом), APNs для push
- **Google Wallet**: Google Wallet API (JWT-based pass creation)
- **Триггер обновлений**: hook в loyalty usecase — при изменении баланса отправить push
- **QR-код на карте**: уникальный идентификатор клиента для сканирования на кассе

#### Ключевые решения

- Apple Developer Account с Pass Type ID и сертификатами — требование Apple
- Google Cloud проект с Wallet API — требование Google
- Хранение push-токенов для обновления карт
- Связь клиента: `phone_number` как основной идентификатор (Wallet-клиент может не иметь Telegram)
- Миграция: добавить `wallet_passes` таблицу с `client_id`, `platform`, `serial_number`, `push_token`

#### Оценка объёма: **XL**

#### Приоритет: **Средний** — расширяет аудиторию, но требует внешних аккаунтов и сертификатов

### 4C. Маркетплейс и оплата внутри бота

**Цель**: Дать клиентам возможность тратить баллы лояльности внутри Telegram-бота — покупать товары, уникальные предложения, оплачивать заказы.

**Бизнес-ценность**: Замкнутый цикл лояльности — баллы не просто накапливаются, а используются, стимулируя повторные визиты.

#### Три сценария (из дизайн-документа)

1. **Маркетплейс в боте** — каталог товаров, покупка за баллы
2. **Уникальные позиции** — обеды дня, «напиток за регистрацию», эксклюзивное меню бота
3. **Оплата через бота** — оплата заказов (интеграция с POS)

Реализация поэтапная — от простого каталога к полноценной оплате.

#### Архитектура

```
internal/
├── entity/
│   └── marketplace.go   # Product, Order, OrderItem
├── usecase/
│   └── marketplace/     # Каталог, заказы, списание баллов
├── repository/
│   └── postgres/
│       └── marketplace.go  # products, orders таблицы
└── controller/
    └── http/group/
        └── marketplace/    # API для админки (управление товарами)
```

- **В боте**: inline-каталог с кнопками, подтверждение заказа, списание баллов
- **В админке**: CRUD товаров, управление ценами в баллах, статистика заказов
- **Интеграция с loyalty**: вызов `SpendPoints` при покупке
- **Оплата (сценарий 3)**: Telegram Payments API или интеграция с POS

#### Ключевые решения

- Баллы vs реальные деньги — разные flow, разные правовые требования
- Товары привязаны к конкретному заведению (bot → org → products)
- Лимиты на трату баллов (настраивается владельцем)
- Сценарий 3 (оплата) может потребовать эквайринг — отдельная юридическая проработка

#### Оценка объёма: **XL** (все три сценария), **M** (только сценарий 1)

#### Приоритет: **Средний** — высокая ценность, но большой объём; начать с каталога за баллы

### 4D. Биллинг и тарифы

**Цель**: Монетизация платформы — подписки для бизнесов с разграничением функций по тарифу.

**Бизнес-ценность**: Основной источник дохода. Дизайн-документ явно указывает: «выбор тарифа НЕ в MVP».

#### Тарифные планы

| План | Назначение | Пример ограничений |
|------|------------|-------------------|
| Trial | Бесплатный пробный период (14-30 дней) | Все функции, ограничение по времени |
| Basic | Малый бизнес, одна точка | До N клиентов, базовые рассылки |
| Pro | Средний бизнес, несколько точек | Расширенная аналитика, сегментация |
| Enterprise | Сети заведений | Без ограничений, SLA, выделенная поддержка |

#### Функции

- Feature gating на основе тарифа (middleware уровня usecase)
- Интеграция с платёжными провайдерами (ЮKassa, CloudPayments)
- Биллинг-дашборд для владельцев (текущий план, история платежей, счета)
- Выбор тарифа при онбординге (шаг 6) или отдельно в настройках
- Автоматическое продление подписки, grace period при неоплате

#### Архитектура

```
internal/
├── entity/
│   └── billing.go        # Tariff, Subscription, Invoice, Payment
├── usecase/
│   └── billing/          # Подписки, платежи, feature gating
├── repository/
│   └── postgres/
│       └── billing.go    # tariffs, subscriptions, invoices, payments
└── controller/
    └── http/group/
        └── billing/      # API для биллинг-дашборда
        └── webhooks/     # Вебхуки от платёжных провайдеров
```

- **Feature gating**: проверка тарифа в usecase-методах, не в middleware HTTP
  ```go
  func (uc *CampaignUsecase) CreateCampaign(...) error {
      if !uc.billing.HasFeature(orgID, "advanced_campaigns") {
          return ErrFeatureNotAvailable
      }
      // ...
  }
  ```
- **Платёжные вебхуки**: отдельный эндпоинт без JWT-авторизации, с подписью провайдера
- **Scheduler**: проверка просроченных подписок, отправка напоминаний

#### Ключевые решения

- ЮKassa vs CloudPayments — выбор провайдера (или оба)
- Рекуррентные платежи (автосписание) vs ручная оплата
- Юридические требования: 54-ФЗ, онлайн-кассы, чеки
- Grace period: сколько дней работает сервис после неоплаты
- Гранулярность feature gating: на уровне модуля или отдельных функций

#### Оценка объёма: **XL**

#### Приоритет: **Высокий** — критично для монетизации, но реализация после набора базы пользователей

### 4E. Расширенные рассылки

**Цель**: Повысить эффективность кампаний через A/B-тестирование, продвинутую аналитику и мультиканальность.

**Бизнес-ценность**: Прямое влияние на конверсию и ROI маркетинговых активностей заведений.

#### Функции

- **A/B-тестирование**: разделение аудитории, сравнение вариантов (тема, контент, время)
- **Аналитика кампаний**: воронки, пути конверсии, heat maps по времени открытия
- **Мультиканальность**: Telegram + push-уведомления (Wallet), будущее — SMS
- **Библиотека шаблонов**: готовые шаблоны рассылок по типам заведений

#### Архитектура

- Расширение существующего `campaigns` usecase
- Новые таблицы: `campaign_variants`, `campaign_results` (для A/B)
- Интеграция с Wallet push (4B) для мультиканальности
- Шаблоны: JSONB-структура в таблице `campaign_templates`

#### Ключевые решения

- Статистическая значимость A/B-тестов — минимальный размер выборки
- Автоматическое определение победителя (по open rate, конверсии)
- Каскадная отправка: сначала Telegram, fallback на push/SMS
- Шаблоны: системные (от Revisitr) + пользовательские

#### Оценка объёма: **L**

#### Приоритет: **Средний** — улучшает существующую функциональность, не критично для запуска

### 4F. Расширенная сегментация

**Цель**: Выйти за рамки RFM — поведенческие и предиктивные сегменты для точечного маркетинга.

**Бизнес-ценность**: Персонализация увеличивает эффективность кампаний и снижает отток клиентов.

#### Функции

- **Кастомные сегменты**: произвольные правила сегментации (возраст, город, сумма заказов)
- **Поведенческие сегменты**: на основе взаимодействий с ботом, предпочтений в меню
- **Предиктивные сегменты**: риск оттока (churn), потенциал upsell
- **Автоматизация**: автоматическое назначение офферов/скидок по сегменту

#### Архитектура

- Расширение существующего `clients` usecase и `segments` модуля (фаза 3)
- Новые таблицы: `segment_rules` (правила кастомных сегментов), `client_predictions`
- Интеграция с данными из POS-интеграций (iiko/r_keeper) для поведенческого анализа
- Scheduler для пересчёта предиктивных сегментов (ежедневно/еженедельно)

#### Ключевые решения

- Алгоритмы предикции: начать с простых эвристик (дней без визита, тренд суммы заказов), позже — ML
- Хранение истории взаимодействий с ботом для поведенческого анализа
- Производительность: пересчёт сегментов для большой клиентской базы
- Privacy: какие данные собирать и как долго хранить (GDPR/152-ФЗ)

#### Оценка объёма: **L**

#### Приоритет: **Низкий** — зависит от объёма данных; имеет смысл при достаточной клиентской базе

## 4. Технологические соображения

### Новые сервисы и инфраструктура

| Компонент | Что добавляется | Влияние на инфраструктуру |
|-----------|----------------|--------------------------|
| Admin-бот (4A) | `cmd/admin-bot` — третий Go-сервис | Новый контейнер в Docker Compose, отдельный Telegram-токен |
| Wallet (4B) | Apple/Google Wallet API клиенты | Apple Developer Account, Google Cloud проект, сертификаты |
| Маркетплейс (4C) | Новые таблицы, расширение bot handlers | Потенциально Telegram Payments API |
| Биллинг (4D) | Платёжные вебхуки, scheduler | Интеграция с ЮKassa/CloudPayments, онлайн-касса |
| Рассылки (4E) | A/B-тесты, мультиканальность | Дополнительная нагрузка на очередь отправки |
| Сегментация (4F) | Предиктивные расчёты | Возможно: отдельный worker для тяжёлых вычислений |

### Общие технические решения

**Shared usecase-ы**: Admin-бот (4A) и маркетплейс (4C) используют те же usecase-интерфейсы, что и `cmd/server`. Выделить usecase-ы в отдельный пакет `internal/usecase`, импортируемый из нескольких `cmd/`.

**Миграции**: каждая подфаза — отдельная группа миграций. Формат не меняется: `YYYYMMDDHHMMSS_description.sql`, goose.

**Docker Compose**: добавление `admin-bot` сервиса. Все сервисы продолжают делить PostgreSQL и Redis.

```yaml
# docker-compose.prod.yml (дополнение)
admin-bot:
  image: ghcr.io/alimbywater/revisitr/admin-bot:latest
  env_file: .env.prod
  depends_on:
    - postgres
    - redis
  restart: unless-stopped
```

**Nginx**: Admin-бот не требует HTTP-роутинга (работает через Telegram API). Wallet может потребовать публичный эндпоинт для скачивания `.pkpass`.

**Мониторинг**: С ростом числа сервисов рассмотреть добавление Prometheus + Grafana для метрик и алертинга.

### Порядок реализации (рекомендация)

```
4D (Биллинг)  ──────────────────►  Монетизация — начать как можно раньше
4A (Админ-бот) ─────────────────►  Уникальная ценность, быстрый цикл
4E (Рассылки)  ─────────────────►  Улучшение существующего
4B (Wallet)    ─────────────────►  Расширение каналов
4C (Маркетплейс) ───────────────►  Начать с каталога за баллы
4F (Сегментация) ───────────────►  Когда достаточно данных
```

Порядок не фиксирован — зависит от приоритетов бизнеса и обратной связи от первых пользователей.
